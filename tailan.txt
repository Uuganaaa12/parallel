		\section{ОРШИЛ}
		Энэхүү хичээлийн курсын ажлын хүрээнд "Дан холбоост жагсаалт дээрх оруулах эрэмбэлэлт" (Insertion sort for singly linked list) - ийн алгоритмыг зэрэгцээ байдлаар хэрэгжүүлж түүний гүйцэтгэлийг цуваа хэрэгжилтийн гүйцэтгэлтэй харьцуулах зорилготой. 
		
		Дан холбоост жагсаалт дээрх оруулах эрэмбэлэлт нь сонгодог "Оруулах эрэмбэлэлт" (Insertion sort) - ээс зангилаа болон заагчид суурилсан өгөгдлийн бүтэц ашигладаг талаарай ялгаатай. Сонгодог алгоритмтай адил массивын элементүүдийг ухрааж шилжүүлэх боломжгүй байдаг. Харин эх жагсаалтаас зангилаа бүрийг ээлжлэн салган авч, шинээр үүсгэж буй эрэмбэлэгдсэн жагсаалтын эхнээс нь эхлэн хайлт хийн, тохирох байрлалыг олох үед заагчуудыг дахин холбох замаар оруулж эрэмбэлдэг. Дан холбоост жагсаалт нь санах ойд дараалж байдаггүй тул кэш санах ойд ээлгүй байдаг. Мөн GPU (Graphic Processing Unit) - ийн зэрэгцээ хэлбэрийн санах ойн хандалт болон SIMT (Single Instruction, Mutiple Threads) загварт тохиромжгүй байдаг тул түүнийг зэрэгцээ байдлаар хэрэгжүүлэх нь тодорхой хэмжээний сорилт болон хүндрэлтэй. 
		
		Ажлын зорилго нь алгоритмыг (1) дараалсан, (2) C++ \texttt{std::thread},
		(3) OpenMP, (4) CUDA архитектурт суурилсан нийт 4 хувилбараар
		хэрэгжүүлж, $n=10^4$, $10^5$, $10^6$ элементүүд дээр гүйцэтгэлийн
		хугацаа, хурдасгалт (SpeedUp), өгөгдөл дамжуулалтын хугацаа (data
		transfer time), achievable performance (throughput) зэрэг
		үзүүлэлтээр харьцуулан шинжлэхэд оршино.
		
		\section{ОНОЛ}
		
		\subsection{Дан холбоост жагсаалт дээрх оруулах эрэмбэлэлт}
		Дан холбоост жагсаалт (\textit{singly linked list}) нь зангилаа
		бүрд утга (\texttt{data}) болон дараагийн зангилаа руу заагч (\texttt{next})
		байрлуулсан, нэг чигийн дараалсан мэдээллийн бүтэц юм. Жагсаалтын
		урт нь $n$ бол элементэд хандах нь $\mathcal{O}(n)$, харин эхэнд
		оруулах/устгах нь $\mathcal{O}(1)$ хугацаатай.
		
		\textit{Оруулах эрэмбэлэлт} (Insertion sort) нь эх жагсаалтыг
		``эрэмбэлэгдсэн'' болон ``эрэмбэлэгдээгүй'' гэсэн хоёр хэсэгт
		хуваан, эрэмбэлэгдээгүй хэсгээс зангилаа авч эрэмбэлэгдсэн хэсгийн
		зөв байрлалд оруулдаг харьцуулалтат алгоритм юм. Жагсаалт хувилбарт
		\textit{шилжүүлэх} (shifting) үйлдлийг \textit{заагч дахин холбох}
		(re-linking) болгож сольдгоороо массив дээрх хувилбараас ялгаатай:
		
		\begin{equation}
			T(n) = \mathcal{O}(n^{2})
		\end{equation}
		
		\noindent учир нь зангилаа бүрд эрэмбэлэгдсэн хэсгийн дунд хүртэл
		дундаж $n/2$ алхам хийгддэг. Memory-н хувьд in-place: $\mathcal{O}(1)$
		нэмэлт орон зайтай.
		
		\subsection{Параллел тооцооллын үндсэн ойлголтууд}
		Параллел тооцоололд нийт даалгаврыг \textit{task} эсвэл \textit{data}
		parallelism зарчмаар жижиг хэсгүүдэд хувааж, олон \textit{thread}
		зэрэг гүйцэтгэдэг. Гол ойлголтууд:
		
		\textbf{Санах ойн шатлал.} CPU нь регистр $\rightarrow$ L1 cache
		$\rightarrow$ L2 $\rightarrow$ L3 $\rightarrow$ DRAM гэсэн шатлалтай.
		Цуваа линкэн жагсаалт дээр заагч дагах (pointer chasing) үед
		зангилаа бүр өөр өөр кэш мөрөнд (cache line) байрладаг тул L1 hit
		rate бага, гүйцэтгэл удаашрана. Жагсаалтыг $T$ хэсэгт хуваавал
		chunk бүр L2-д сайн багтаж кэш миссийг бууруулдаг — энэ нь параллел
		хувилбарт \textit{linear speedup-аас давсан хурдасгалт} (super-linear
		speedup) өгдөг гол шалтгаан болдог.
		
		\textbf{Зангилаа (thread) ба синхрончлол.} Хуваагдсан хэсэг хооронд
		ажил дуусгахдаа \texttt{join}, \texttt{barrier}, \texttt{atomic}
		зэрэг механизмаар синхрончлогдох шаардлагатай. Pairwise merge нь
		шатлал бүрийн төгсгөлд барьерийг шаарддаг.
		
		\textbf{SIMT (CUDA).} GPU нь warp буюу 32 thread-ийг нэгэн зэрэг
		ижил инструкци гүйцэтгүүлдэг. Тиймээс \textit{branch divergence}
		багатай, coalesced memory access-тэй кодыг сайн дэмждэг. Pointer-based
		linked list нь random access pattern-той тул GPU-д шууд бичих нь
		үр ашиггүй.
		
		\subsection{std::thread, OpenMP, CUDA загварууд}
		
		\textit{C++ \texttt{std::thread}.} C++11-аас хойшхи стандарт сан.
		Thread бүрийг гар аргаар үүсгэж, \texttt{join}-оор хүлээдэг. Гар
		удирдлагатай тул maximum уян хатан боловч boilerplate их.
		
		\textit{OpenMP.} Compiler директивт суурилсан share-memory
		параллел загвар. \texttt{\#pragma omp parallel for} зэрэг pragma-ар
		loop-уудыг параллел болгодог. Thread pool, scheduling-ийг runtime
		автоматаар хариуцна.
		
		\textit{CUDA.} NVIDIA GPU-н дээрх параллел загвар. Грид
		(\texttt{Grid}) нь олон \texttt{Block}-аас, block нь олон
		\texttt{Thread}-аас бүрдэнэ. Host (CPU) нь kernel-ийг launch хийж,
		\texttt{cudaMemcpy}-аар санах ой Host~$\leftrightarrow$~Device хооронд
		дамжуулна. Memory hierarchy: register $\rightarrow$ shared memory
		(per-block, $\le 48\,$KB) $\rightarrow$ global memory (DRAM).
		
		\section{АЛГОРИТМЫН ЗОХИОМЖ}
		
		\subsection{Дараалсан (Sequential) хувилбар}
		Энгийн pointer-based insertion sort. Гадаад while-loop нь зангилаа
		бүрийг авч, дотоод while-loop нь \texttt{sorted} жагсаалтын зөв
		байрлалыг хайж, заагчуудыг шинэчилнэ. Memory access pattern нь
		\textit{санамсаргүй} (random) тул L1/L2 кэш miss их үүснэ.
		
		\begin{figure}[H]
			\centering
			\includegraphics[width=\linewidth]{assets/diagram_sequential}
			\caption{Sequential insertion sort-ийн блок схем.\\
				\textit{Тайлбар:} Дээд талд эх жагсаалтын head, дараа нь
				curr/next loop, доор нь эрэмбэлэгдсэн ``sorted'' жагсаалт.
				Заагчуудыг дахин холбохдоо элементийн бодит шилжилт хийдэггүй
				(re-linking only).}
		\end{figure}
		
		\textit{Псевдо код 1 --- Sequential Linked-List Insertion Sort:}
		
\begin{lstlisting}
function insertionSortLL(head):
	sorted <- NULL
	curr   <- head
	while curr != NULL do
		next <- curr.next
		if sorted == NULL or sorted.data >= curr.data then
			curr.next <- sorted
			sorted    <- curr
		else
			p <- sorted
			while p.next != NULL and p.next.data < curr.data do
			p <- p.next
		curr.next <- p.next
		p.next    <- curr
		curr <- next
	return sorted
\end{lstlisting}
		
		Дотоод цикл дунджаар $n/2$ алхам хийгддэг тул нийт хугацаа
		$\mathcal{O}(n^2)$ болно.
		
		\subsection{std::thread хувилбар}
		\textbf{Гол санаа:} жагсаалтыг $T$ тэнцүү хэсэгт \textit{бодит
			заагчийг таслан} (\texttt{curr->next = nullptr}) хувааж, thread бүр
		өөрийн хэсгийг pointer-based insertion sort-оор эрэмбэлнэ. Эцэст нь
		$\log_{2}T$ шатлалтай pairwise merge-ийг параллел гүйцэтгэнэ.
		
		\textit{Хугацааны нийлмэл (онол, $T$ thread):}
		\begin{equation}
			T_{\text{par}} \approx \mathcal{O}\!\left(\frac{n^2}{T} + n\log T\right)
		\end{equation}
		Үүнд $n^2/T$ нь chunk-уудын паралелл insertion sort-ийн хугацаа,
		$n \log T$ нь pairwise merge-ийн нийт ажил.
		
		\begin{figure}[H]
			\centering
			\includegraphics[width=0.98\linewidth]{assets/diagram_stdthread.png}
			\caption{\texttt{std::thread} хувилбарын архитектур: split $\to$
				parallel sort $\to$ parallel pairwise merge.\\
				\textit{Тайлбар:} Эх жагсаалт $T=4$ chunk-д хуваагдаж, thread бүр
				insertion sort хийнэ. Дараа нь хосоор merge хийх 2 шатлал.}
		\end{figure}
		
		\textit{Псевдо код 2 --- std::thread Parallel Insertion Sort:}
		\begin{lstlisting}
function parallelInsertionSort(head, n, T):
	chunks <- splitList(head, n, T)   // size = T
	parallel for i = 0 .. T-1 do
		chunks[i] <- insertionSortLL(chunks[i])
	while size(chunks) > 1 do
		next <- empty array of size ceil(|chunks|/2)
		parallel for i = 0 .. |chunks|/2 - 1 do
		next[i] <- mergeSortedLL(chunks[2i], chunks[2i+1])
		if |chunks| is odd then
			next[last] <- chunks[last]
		chunks <- next
	return chunks[0]
		\end{lstlisting}
		
		\subsection{OpenMP хувилбар}
		Дээрх алгоритмтай адил, ялгаа нь параллел loop-ыг \texttt{\#pragma
			omp parallel for schedule(static) num\_threads(T)} директивээр
		тэмдэглэдэгт. OpenMP runtime нь thread pool-ийг автоматаар
		үүсгэдэг тул thread create/join-ийн overhead багасна.
		
		\begin{figure}[H]
			\centering
			\includegraphics[width=0.98\linewidth]{assets/diagram_openmp.png}
			\caption{OpenMP fork--join загвар: \texttt{\#pragma omp parallel for}
				директивийн ажиллагаа.\\
				\textit{Тайлбар:} Master thread fork хийж $T$ worker үүсгэнэ.
				Loop-ын төгсгөлд implicit barrier байх ба бүх thread join хийгдэнэ.}
		\end{figure}
		
		\textit{Псевдо код 3 --- OpenMP Parallel:}
		\begin{lstlisting}
function parallelInsertionSortOMP(head, n, T):
	chunks <- splitList(head, n, T)
	#pragma omp parallel for schedule(static) num_threads(T)
	for i = 0 .. T-1 do
		chunks[i] <- insertionSortLL(chunks[i])
	while size(chunks) > 1 do
		pairs <- |chunks| / 2
		#pragma omp parallel for schedule(static)
		for i = 0 .. pairs-1 do
		next[i] <- mergeSortedLL(chunks[2i], chunks[2i+1])
		if odd(|chunks|) then next[last] <- chunks[last]
			chunks <- next
	return chunks[0]
		\end{lstlisting}
		
		\subsection{CUDA хувилбар}
		GPU дээр pointer-based linked list нь \textit{санамсаргүй global
			memory access} үүсгэдэг тул шууд ажиллахад үр ашиггүй. Тиймээс
		дараах гурван үе шаттай стратегийг ашигласан:
		\begin{enumerate}\itemsep 0pt
			\item \textbf{Host: List $\rightarrow$ Array.} Жагсаалтыг $\mathcal{O}(n)$
			хугацаанд массив руу хөрвүүлэх.
			\item \textbf{H2D + GPU sort.} \texttt{cudaMemcpy} ашиглан Device-руу
			хуулж, kernel-уудаар эрэмбэлнэ:
			\begin{itemize}\itemsep 0pt
				\item \texttt{insertionSortKernel}: block бүр 1024 элементтэй
				chunk-аа shared memory дотор insertion sort хийнэ.
				\item \texttt{mergeKernel}: $\log_{2}(\text{numChunks})$
				шатлалтай ping-pong (\texttt{d\_a} $\leftrightarrow$
				\texttt{d\_b}) parallel merge.
			\end{itemize}
			\item \textbf{D2H: Array $\rightarrow$ List.} Үр дүнг Host-руу буцаан
			хуулж, дахин дан холбоост жагсаалт болгоно.
		\end{enumerate}

			\includegraphics[width=0.98\linewidth]{assets/diagram_cuda.png}
			\caption{CUDA архитектур --- Grid / Block / Thread шатлал ба санах
				ойн зохион байгуулалт.\\
				\textit{Тайлбар:} Зүүн талд Host-аас Device-руу cudaMemcpy. Баруун
				талд Grid дотор олон Block, block бүр 32 thread, shared memory.

		\textit{Псевдо код 4 --- CUDA Insertion Sort Kernel:}
		\begin{lstlisting}
__global__ insertionSortKernel(arr, n, chunkSize):
	chunkId <- blockIdx.x
	start   <- chunkId * chunkSize
	end     <- min(start + chunkSize, n)
	if threadIdx.x == 0 then
	for i = start+1 .. end-1 do
		key <- arr[i]
		j   <- i - 1
		while j >= start and arr[j] > key do
			arr[j+1] <- arr[j]
			j <- j - 1
		arr[j+1] <- key

__global__ mergeKernel(in, out, n, width):
	idx   <- blockIdx.x * blockDim.x + threadIdx.x
	start <- idx * 2 * width
	if start >= n then return
	mid <- min(start + width,    n)
	end <- min(start + 2*width,  n)
	merge(in[start..mid], in[mid..end]) -> out[start..end]
		\end{lstlisting}

		\texttt{sizeof(int)}~$=4$ байт. Иймд нэг чигт дамжуулсан хэмжээ:
		\begin{equation*}
			4 \times 10^6 \text{ байт} = 4{,}000{,}000 \text{ байт}
			\;=\; 4\,\text{MB} \;=\; 3.815\,\text{MiB}.
		\end{equation*}
		\noindent Хоёр чигт ($H{\to}D$ + $D{\to}H$) нийт $8\,\text{MB} =
		7.629\,\text{MiB}$. (Тэмдэглэл: $1\,\text{MB}=10^6$ байт, SI стандарт;
		$1\,\text{MiB}=2^{20}=1{,}048{,}576$ байт, IEC binary.)
		
		\textit{Effective bandwidth (хэмжилтээс):}
		\begin{equation*}
			\text{H2D:}\quad
			\frac{4 \times 10^6 \text{ байт}}{0.960 \times 10^{-3} \text{ с}}
			\;\approx\; 4.17\,\text{GB/s} \;\;(\equiv 3.88\,\text{GiB/s}),
		\end{equation*}
		\begin{equation*}
			\text{D2H:}\quad
			\frac{4 \times 10^6 \text{ байт}}{1.048 \times 10^{-3} \text{ с}}
			\;\approx\; 3.82\,\text{GB/s} \;\;(\equiv 3.55\,\text{GiB/s}).
		\end{equation*}
		
		Туршилт Google~Colab-ийн NVIDIA Tesla~T4 GPU дээр явсан тул
		PCIe~Gen3~x16 интерфэйсээр явсан болно. PCIe~Gen3~x16-ийн онолын
		хязгаар $\sim 15.75$~GB/s, реал \texttt{cudaMemcpy} bandwidth
		ихэвчлэн 5--12~GB/s. Хэмжсэн 3.82--4.17~GB/s нь pinned memory
		ашиглаагүй (\texttt{cudaMallocHost} биш ердийн \texttt{malloc})
		тохиолдолд хүлээх хүрээнд буюу 25--40\% theoretical хязгаараас.
		Pinned memory ашигласан бол $\sim 12$~GB/s хүртэл өсөх боломжтой
		байсан --- цаашдын оптимизацийн боломж.
		
		\subsection{Анализ}
		
		\textbf{(1) Жижиг өгөгдөл ($10^4$).} \texttt{std::thread} нь OpenMP-ээс
		бага зэрэг хурдан (1.244 vs 1.287 мс), CUDA нь дараалснаас
		1.25$\times$ хурдан (27.57 мс vs 34.35 мс) боловч CPU параллелаас
		22 дахин удаан. Шалтгаан: $\text{chunkSize}=1024$ үед $n=10^4$ дээр
		ердөө 10 block үүсэх тул GPU-н хэдэн зуун SM-ийн ердөө 1--2\%
		ашиглагдана. Grid size нь bottleneck.
		
		\textbf{(2) Дунд өгөгдөл ($10^5$).} CPU параллел хоёр хувилбар
		\textbf{200--211$\times$} хурдасгалттай — $T=10$ thread-аас давсан
		энэ ``super-linear'' үр дүн нь гол шалтгаан 2: (а) chunk-ууд L2
		кэшэд сайн багтсан тул pointer chasing-н кэш миссийг бууруулсан;
		(б) $n^2/T$ зүйлийн нийлмэл бус жижигтсэнг тооцвол $T \cdot$ кэш
		сайжралт $= \sim$20 дахин нэмэлт олз. CUDA $n=10^5$ дээр аль
		хэдийн өрсөлдөгч (176$\times$).
		
		\textbf{(3) Том өгөгдөл ($10^6$).} CPU параллел SpeedUp 92$\times$
		болж буурсан --- кэш сайжралт нь 1M дээр L2 ($\sim 10$~MB) хүчин
		чадлаас давсан тул эффект буурсан. Харин CUDA \textbf{4849$\times$}
		хүрсэн — GPU-н минглэн thread параллелизм энэ хэмжээнээс л
		жинхэнэ ажилладаг ($\sim$1000 block, бүх SM ажиллана).
		
		\textbf{(4) Data transfer time.} CUDA-ийн нийт хугацааны 0.5\%-аас
		бага (1M дээр 0.96+1.05 $= 2.01$ мс $/ 406$ мс $= 0.50\%$) —
		өгөгдөл дамжуулалт нь bottleneck биш. Гол non-compute overhead нь
		\texttt{array$\rightarrow$list} (38.85 мс, нийт хугацааны 9.6\%) —
		Host-ийн \texttt{new} malloc-ийн ажиллагаа.
		
		\textbf{(5) Total operations.} Sequential 1M-н хувьд $\sim 5 \cdot
		10^{11}$ харьцуулалт хийсэн. Параллел хувилбар $T=10$ дээр $\sim
		n^2/T + n \log T \approx 10^{11}$ харьцуулалт. CUDA дээр
		chunkSize=1024 учраас $\frac{1024^2}{2} \cdot \frac{n}{1024} = 512
		\cdot n = 5.12 \cdot 10^8$ insertion-sort-ийн compare + $n \log_2
		\frac{n}{1024} \approx 10^7$ merge compare $= \sim 5.2 \cdot 10^8$
		нийт. Тийм тоог 343.85 мс-д хийж байгаа нь $\sim 1.51$~Gops. 
		
		\section*{ДҮГНЭЛТ}
		
		Энэхүү судалгааны ажлаар дан холбоост жагсаалт дээрх оруулах
		эрэмбэлэлтийг дөрвөн хувилбараар (Sequential, \texttt{std::thread},
		OpenMP, CUDA) хэрэгжүүлж, $10^4$, $10^5$, $10^6$ элементийн
		санамсаргүй өгөгдөл дээр харьцуулсан туршилт хийсэн болно.
		Туршилтын үр дүнгээс дараах гол дүгнэлтүүд гарч байна:
		
		(1) \textbf{Параллел стратеги бодитоор үр дүнтэй.} ``Split $\to$
		parallel sort $\to$ pairwise merge'' стратегиар дөрвөн хувилбарын
		аль нь ч 100$\times$-ээс илүү хурдасгалт өгсөн. Энэ нь зөвхөн thread
		тоонд биш, кэш локаль байдал сайжирсан үр дагавар.
		
		(2) \textbf{std::thread vs OpenMP.} Хоёр хувилбар бараг ижил
		гүйцэтгэлтэй ($10^5$, $10^6$ дээр $\pm 1\%$). $10^4$ дээр
		\texttt{std::thread} 27\%-аар хурдан --- OpenMP-ийн runtime
		initialization overhead жижиг өгөгдөлд мэдрэгддэг. Реал төсөлд
		OpenMP-ийн pragma загвар нь кодын уншилт сайжруулдаг тул илүү
		тохиромжтой.
		
		(3) \textbf{CUDA-ийн ``crossover point''.} CUDA нь $n=10^4$ дээр
		дараалснаас 1.25$\times$ хурдан боловч CPU параллелаас 22$\times$
		удаан (grid 10 block, GPU underutilized). $n=10^5$ дээр аль хэдийн
		өрсөлдөгч (176$\times$), $n=10^6$ дээр CPU параллелаас \textbf{52
			дахин} хурдан (4849$\times$). GPU acceleration нь $\gtrsim 10^5$
		хэмжээнээс эхлэн үр дүнтэй болохыг тодорхой харуулсан.
		
		(4) \textbf{Data transfer нь bottleneck биш.} 1M элемент дээр
		H2D+D2H нийт $2.01$~мс --- нийт хугацааны $0.50\%$ ($\ll 1\%$).
		Энэ нь PCIe~Gen3~x16 (онолын $\sim 16$~GB/s)-ийн ердийн bandwidth
		дээр 8~MB өгөгдөл маш бага хугацаа авдгийг харуулна. Гол
		bottleneck нь \textbf{GPU-ийн compute} ($343.85$~мс, $84.7\%$),
		хоёрдугаарт Host дээр массивыг буцаагаад зангилаа болгож
		хувиргах \texttt{new Node} malloc loop ($38.85$~мс, $9.6\%$). Энэ
		нь pointer-based мэдээллийн бүтцийн нийтлэг асуудал --- GPU-ын
		дараах $\mathcal{O}(n)$ host-side rebuilding нь шинэ bottleneck
		болж гарч ирэв.
		
		
		(5) \textbf{Amdahl-ийн хууль ажиллаж байна.} CPU параллел SpeedUp
		$10^5$ дээр пик 200$\times$ хүрч $10^6$ дээр 93$\times$-д буурсан
